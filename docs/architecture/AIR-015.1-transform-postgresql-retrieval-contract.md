# AIR-015.1: Transform PostgreSQL Retrieval Contract

**Issue**: #128 | **Epic**: #127 (AIR-015)

## The Problem

Right now, extract creates raw data in memory, passes it directly to transform, then saves it to the database. We want transform to query the database instead—so extract and transform aren't tightly coupled.

**Current**: 
```
Extract → (in-memory records) → Transform → Save to DB
```

**Target**: 
```
Extract → Save to DB → Transform queries DB → Load gold data
```

## What We're Doing

### 1. Use `pipeline_run_id` as the Query Key

Transform will fetch raw data using `pipeline_run_id`—a simple integer that uniquely identifies one pipeline execution.

**Why?**
- It's already a foreign key in the raw responses table (indexed for speed)
- One query gets all raw data for the entire run
- Clean, simple, no extra lookups needed

### 2. Query All Rows for the Run

```sql
SELECT * FROM raw_air_pollution_responses 
WHERE pipeline_run_id = ?
ORDER BY city_id, request_start_utc
```

That's it. Get all raw responses for the run, in a consistent order.

### 3. Handle Missing or Incomplete Data

| What Happens | How We React |
|---|---|
| No rows found | Log it and continue (transform gets empty input, returns no gold rows) |
| Some cities failed | Process what we have (same as extract stage already does) |
| Bad HTTP response in the data | Skip it (transform already ignores failed responses) |

### 4. Put Retrieval Logic in Its Own Module

Create `pipeline/repository/raw_air_pollution.py` with a single function:

```python
def load_raw_responses_by_pipeline_run_id(pipeline_run_id: int) -> list[RawAirPollutionRecord]:
    """Fetch all raw responses for a pipeline run from the database."""
```

**Why a separate module?**
- Keeps DB queries away from orchestration code
- Makes it easier to test transform independently
- One place to maintain all DB access for raw data

## Code Changes

**New module**: `services/pipeline/src/pipeline/repository/raw_air_pollution.py`

**Modified**:
- `run_transform_stage()` in orchestration — now takes `pipeline_run_id` instead of raw records
- Orchestration calls the repository function to load data

**Unchanged**:
- `build_gold_from_raw_records()` stays the same (takes records, returns gold DataFrame)
- Transform logic doesn't change

## Testing

Old tests can still pass records directly to `build_gold_from_raw_records()`.

New tests will:
1. Create test data in the database
2. Pass `pipeline_run_id` through orchestration
3. Verify gold output matches what we'd get from in-memory records
